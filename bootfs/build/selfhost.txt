# RYMOS self-hosting status

name: RYMOS Rust self-hosting
milestone: 8
goal: Run a Rust toolchain on RYMOS after the OS grows enough POSIX-like runtime support.

summary: blocked=1, partial=11

[partial] Program ABI
  ELF programs run synchronously in trusted kernel mode with ABI v23, argv-preserving spawn_argv, argv-style reads, process-local environment overrides, parent/child tracking, consuming wait/wait_any status, synchronous spawn with inherited std fds, pipe-fed child stdin, separate stdout/stderr capture, descriptor IO, stdio fds, pipes, dup2 redirection, a std_fd query for the current std-fd redirection target (v23), stat, listing, mkdir, unlink, rename, cwd/path resolution, errno-style last_error, environment reads, process status, heap and mmap-style page allocation, and calibrated monotonic ticks (see Time and OS Services). stat/list now also report created/modified tick timestamps and a permission-style mode byte. A cargo-shaped smoke test (cargolike/bigoutput) found two real ceilings too tight for a real cargo-style child: the per-pipe buffer (APP_PIPE_BUFFER_SIZE) silently truncated captured child stdout past 1 KiB, raised to 8 KiB; the per-process env table (APP_ENV_COUNT) was 8 slots total including 6 kernel defaults (only 2 free), and exhausting it panicked std::env::set_var into a whole-process abort rather than just failing the call (panic_abort, no unwinding) -- raised to 64 slots, re-verified the raised ceiling still fails the same safe way (caught by category 3's exception handler) once genuinely exhausted. Confirmed not an issue: repeated sequential spawns (8 in a row via Command::output()) don't leak process-table slots. A follow-up nested-spawn test (relay, which spawns itself recursively rather than sequentially, recreating a real cargo -> rustc -> linker chain) found and fixed three real, layered bugs: (1) each Command::output() call holds 3 pipe slots open for its entire duration including a nested child's runtime, so APP_PIPE_COUNT (4) supported barely more than one link of nesting before ERR_NOSPC -- raised to 12; (2) the real, deeper bug: reset_stdio (in both rymos-user and sys::process::rymos) unconditionally reset STDIN/STDOUT/STDERR back to the real console after a Command call, correct only when the caller's own ambient stdio already was the console -- wrong for a nested call, whose caller's stdout is itself someone else's capturing pipe, confirmed to silently drop a nested child's output straight to the console instead of its actual caller's pipe; fixed with the new std_fd ABI call (v23) so both Command implementations save the real pre-redirect value and restore exactly that; (3) app_restore_after_spawn used to snapshot the entire live APP_PIPES array into a local before merging, an extra full-size copy on top of the already-large AppStateSnapshot stacked at every nesting level -- this kernel has no stack guard page, so the overflow silently corrupted memory and hung (not crashed) a deep chain; fixed by merging the live pipe data directly into the snapshot's own field instead of a second full-array copy. Verified live in QEMU: a 3-level nested relay chain and a 4-level chain through cargolike's std::process::Command both complete cleanly with every level's output correctly propagated to the top.
[partial] Core Runtime
  rymos-user exposes console, raw args, argv reads, mutable env wrappers, pid, line input, read-only BootFS reads, and a stdish compatibility shim for fs/env/process/io/time/path/temp-dir/error basics.
[partial] Persistent Filesystem
  RYMFS5 persists 256 MiB files across up to 4 non-contiguous extents per file, 256 entries with 96-byte names, compact nested directories, unlink/rename, append/create-new open modes, OpenOptions-style runtime wrappers, per-entry created/modified tick timestamps plus a permission-style mode byte, and sparse writes (seeking past the current end and writing zero-fills the gap instead of leaving stale disk contents). Growth coalesces into the last extent when the newly-found free run is physically contiguous with it, so ordinary sequential writes don't burn through the 4-extent budget on nothing but adjacent sectors. Verified live in QEMU: a file too big to fit in a single freed hole between two other files split cleanly across two extents and read back byte-exact; a path well past the old 30-byte limit, a sparse write with a zero-filled gap, and 20 concurrent open file descriptors all round-tripped correctly. Still open: the 256-entry/96-byte-name ceilings are fixed compile-time limits, not true unbounded growth -- that would mean moving the entry table itself onto growable disk extents (the header is read into a stack-local buffer in a few places today, which is the actual limiter on how far a fixed bump can safely go).
[partial] Process Model
  PIDs, parent PID tracking, exit status, consuming ABI wait/wait_any, and synchronous spawn with inherited std fds exist. Spawned children now run in their own isolated address space (see Memory Management), which removed the old fixed-address snapshot/restore-ELF dance and fixed a real bug where a child's execution silently corrupted the parent's writable globals. A real reaping bug is now fixed: a brand-new spawn used to silently reuse any Exited/Failed process-table slot regardless of whether its real parent had ever called wait/wait_any on it, destroying a genuine child's exit status before it was ever collected; slot reuse now requires the zombie to already be waited on (process_find_reapable_slot), and top-level console `run`'d processes (which have no ABI parent to wait on them) are marked pre-reaped so they don't become permanent zombies. A genuinely deferred/queued spawn (enqueue at spawn time, actually run later when something waits) was attempted and reverted after it broke live in QEMU: several of rysh's own shell built-ins, and rymos-user's Command helpers, redirect stdio/cwd/env, spawn, and read the result with no intervening wait() call at all, relying entirely on spawn's synchronous completion -- deferring it left a redirected pipe disconnected from ever being read. That postmortem named the real precondition for concurrency -- the ABI's flat globals moved into a real per-process control block -- and that part is now done: all per-process ABI state (console/bootfs excluded, since those are genuinely process-independent; fds/pipes excluded too, deliberately left as a still-shared table, a disclosed and separately-scoped limitation) lives directly on Process now, addressed via one CURRENT_PROCESS_INDEX global instead of 18 flat statics swapped in and out via a stack-resident snapshot struct on every spawn. Verified live in QEMU: the full existing regression suite, including nested-spawn tests, produced byte-identical output before and after this migration. Concurrent (actually parallel/interleaved) children still need a real scheduler: there is still no saved per-process CPU register/stack context, no per-process kernel stack, and no preemption, so execution is still a synchronous nested call -- this remaining work (context-switch assembly, a cooperative scheduler, then real preemption via a timer interrupt) was deliberately paused before starting, given its size and risk, rather than pushed through in one sitting.
[partial] Memory Management
  UEFI memory map, physical page allocation, kernel-owned PML4, virtual mappings, per-process heap windows, guarded mmap-like map/unmap regions, and process-exit unmap/reclaim exist. Spawned children now get their own private PML4 for the fixed program-image window (0x200000-0x9000000): a shallow top-level PML4 clone plus selectively-privatized PDPT/PD entries just for that range, so the kernel's own mappings and every process's heap/mmap windows stay shared while each process's code/data is genuinely isolated. destroy_process_address_space reclaims the private PT/PD/PDPT/PML4 structural pages plus any data pages still mapped there, so this doesn't leak a page-table page per spawn. Heap/mmap page-table page reclaim is now done too: reclaim_process_window_tables frees the PT pages (heap) or PD+PT pages (mmap) exclusively owned by an exiting PID's window, safe without a private-PML4 walk because each PID's window sits at a fixed, alignment-guaranteed address (a heap window is exactly a quarter of one shared PD; a mmap window is exactly one whole, exclusively-owned PD) purely as a function of its own never-reused PID. Verified live in QEMU: three consecutive spawn/exit cycles of the same program leave the physical page allocator's used-page count exactly flat, not growing per cycle. The kernel also now has its first CPU exception handling: an IDT (built without a new GDT -- it reads whichever code-segment selector UEFI's firmware already set up) with hand-written naked-function stubs (the x86-interrupt ABI is nightly-only; this kernel builds on stable) for all 32 exception vectors turns a fault -- a guard-page touch, a null-pointer access, a divide-by-zero, an invalid opcode -- into a clear serial diagnostic (vector, mnemonic, error code, CR2 for page faults, faulting RIP, current PID/process name) followed by a clean halt, instead of a silent, undiagnosable QEMU reset. Verified live in QEMU across guard-page (both before and after the mapped region), raw hardware divide-by-zero, and invalid-opcode faults via a new `faultcheck` program (deliberately not part of the automated regression, since a passing run halts the machine). Found and disclosed, not fixed, along the way: a null-pointer write does *not* fault today, because address 0 is already mapped as part of the low-memory identity mapping rather than being guarded -- a real, separate memory-layout gap. Also disclosed: recovering from a fault (killing just the offending process, not the whole OS) needs a prepared non-local-exit point to unwind an arbitrary nested Rust call stack, and a double fault caused by genuine kernel stack exhaustion could still overrun into a real triple fault without a dedicated IST/TSS stack for it -- both separate, larger pieces of work, not attempted here. Assessed (not changed): per-process heap (256 MiB) and mmap (768 MiB) window sizes are already far larger than what fits in a 256 MiB QEMU test VM once the 142 MiB fixed program-image reservation is accounted for -- the real ceiling today is available physical RAM and that reservation, not the virtual window sizes, so raising the latter wouldn't currently buy anything. Still open: relocatable/PIE loading (every process still uses the same virtual address for its image, just backed by different physical pages -- this is isolation via separate tables, not relocation).
[partial] File Descriptor ABI
  BootFS read handles, RYMFS5 read/write/append/seek handles, stat, listing, compact nested mkdir, unlink, rename, 256 entries, 256 MiB files, per-entry timestamps/mode, non-contiguous (up to 4 extents) allocation per file, sparse writes, and up to 32 concurrent per-process file descriptors (raised from 8) exist.
[partial] Toolchain Target
  x86_64-rymos.json now parses on current nightly (target-pointer-width/target-c-int-width as integers, explicit rustc-abi=softfloat, max-atomic-width, and singlethread=true) and RYMOS_TARGET_MODE=custom plus nightly rust-src builds real programs end to end via `-Z build-std -Z json-target-spec`. A forked rust-lang/rust checkout also lives at toolchain/rust (git submodule) with a patched library/std for real std support; see the libstd entry.
[partial] libcore
  core, alloc, and compiler_builtins now build and link for x86_64-rymos via nightly build-std; verified end to end by building and installing the hello program through scripts/rymos-sdk.py's custom target mode.
[partial] liballoc
  rymos-user grows a bump heap through kernel-mapped pages, exposes guarded mmap-like regions, and can run Vec/String smoke tests in per-process virtual windows; mapped data pages are unmapped and reclaimed on process exit.
[partial] libstd
  A real std program now boots and runs on x86_64-rymos: toolchain/rust (a forked rust-lang/rust checkout, built locally via ./x.py build library/std --stage 1) adds sys::pal::rymos with a real _start(abi) -> i32 entry that stores the ABI pointer and calls the compiler-generated main() from #[lang = "start"] (same pattern as Hermit's runtime_entry), a real bump allocator in sys::alloc::rymos over mem_alloc_pages, and real Stdout/Stderr writes through the ABI console call in sys::stdio::rymos. Category 5 (Rust target and libraries) landed real implementations, not just stdio/alloc, for most of the rest: sys::env (real getenv/setenv/unsetenv/env() against the ABI's env_get/env_list/env_set/env_remove, no local cache needed since the kernel is the source of truth), sys::args (real argv via argv_count/argv_get -- queried live on every call rather than cached at start, since unlike env vars argv never changes after a process starts), sys::paths (real getcwd/chdir/temp_dir against the ABI's cwd/chdir/TMPDIR env var), sys::fs (a real File plus stat/list/mkdir/unlink/rename-backed directory operations; RYMOS has no symlinks/hardlinks/file locking/permission-setting so those honestly stay unsupported rather than faking them), sys::io::error (real ERR_* to ErrorKind mapping via the ABI's last_error, replacing the old generic fallback that always claimed success), sys::random (RDRAND when CPUID reports it, falling back to a disclosed-non-cryptographic TSC-seeded PRNG instead of panicking), and sys::time's Instant and SystemTime (both fully real now that category 4 calibrated time_ticks and added time_unix_nanos -- see Time and OS Services; duration arithmetic and SystemTime::now() both work, verified live in QEMU). std::process::Command is real now too (category 6): sys::pipe::rymos wraps the ABI's pipe/read/write_fd/close, and sys::process::rymos implements Command/Stdio/Process/ExitStatus using the same snapshot-mutate-then-restore-around-the-synchronous-spawn pattern rymos-user's own Command already used successfully (dup2 the child's stdio onto the ambient std fds, chdir/env_set overrides, spawn, restore everything). Because RYMOS's spawn runs the child to completion inline before returning, Command::output()/status() work correctly and completely (the child's exit status and full stdout/stderr are already final by the time these methods read them) -- exactly the pattern cargo/rustc invocation actually uses. The one disclosed, real limitation: Command::spawn() followed by writing to Child::stdin does not behave like a real OS, since the child already ran (and already read whatever was in its stdin pipe, or read nothing) by the time spawn() returns -- fixing that needs real concurrent execution (category 2's still-open scheduler work), not more ABI wiring. Stdio::null() has no real /dev/null to back it, so it uses a scratch pipe whose unused end is dropped (reading an empty, nothing-else-attached pipe returns 0/EOF immediately rather than blocking, so this behaves exactly like a null device both ways). Verified live in QEMU: stdreal's Command::output() captured a real child's exit code and 326 bytes of real stdout; Command::status() with an env() override succeeded and the override was confirmed not to leak into the parent's own environment afterward. std::process::id() is real via the ABI's pid call. stdreal also has a real, repeatable build pipeline now instead of being a hand-built one-off: RYMOS_TARGET_MODE=std python3 scripts/rymos-sdk.py install stdreal builds it via the rymos-fork toolchain's -Z build-std path, automatically clears the stale build-std target cache first (the exact gotcha that caused a real, confirmed false crash earlier this session), recreates the rust-lld symlink the stage1 sysroot loses on every toolchain rebuild, and strips debug symbols after -- the same manual sequence docs/dev-environment.md documents, now scripted into one command rather than remembered by hand. Deliberately kept out of rymos-packages.toml/install-all's default flow (which always uses the stable/custom no_std toolchain), since std mode is never auto-selected and only stdreal opts into it today. Verified live in QEMU with a new stdreal program (a genuine #![feature(restricted_std)] binary, not routed through rymos-user) exercising pid, real argv, env get/set/iterate/remove, cwd/temp_dir, file write/read_to_string/exists/read_dir, Instant ordering, and a HashMap (exercising the new random support) -- all correct, alongside the full existing no_std regression suite. Found and fixed along the way: the bootloader's hardcoded 4 MiB INITRD_READ_LIMIT silently truncated bootfs once a debug std binary's size exceeded it (raised to 32 MiB, still relevant); a stale `-Z build-std` target cache silently kept using pre-edit sys module code after a fresh source change, masquerading as unrelated crashes until `rm -rf target/x86_64-rymos` forced a real rebuild -- exactly the gotcha already flagged in docs/dev-environment.md, hit for real this time. std::thread::sleep is also real now (category 4's sleep_nanos), verified via a real 15ms sleep measured against real Instant duration arithmetic. Still open: real thread spawning (not needed while RYMOS stays single-threaded per process; sleep needed no spawning support to be real). FileAttr::modified()/created() are also real now, not unsupported() stubs: Stat already carried created_ticks/modified_ticks (category 5), but the on-disk PFS values were raw rdtsc cycles, not the calibrated ns-since-boot unit time_ticks's ABI call reports -- userspace never sees the kernel-internal TSC_HZ/BOOT_TSC needed to convert raw cycles back to real time, so the kernel now stores ns_since_boot() there instead (pfs_set_entry/pfs_touch_modified in kernel/src/main.rs), and sys::fs::rymos::FileAttr::modified()/created() convert it to a real SystemTime via time_unix_nanos() - time_ticks() as the boot-time offset. accessed() stays unsupported() since Stat has no real access-time field. Verified live in QEMU with the cargolike test program: fs::metadata(...).modified() on a freshly-written PFS file returned a real, plausible Unix timestamp. Nested std::process::Command usage (a spawned child that itself spawns, e.g. cargo -> rustc -> linker) is also real now: sys::process::rymos's reset_stdio used to unconditionally reset stdio back to the console after a Command call, which silently broke a nested caller's own redirection; fixed via the new std_fd ABI call (v23) that lets it save and restore the caller's actual pre-redirect target instead of assuming console. Verified live in QEMU: a 4-level nested Command chain through cargolike completes cleanly with every level's output correctly propagated to the top -- see Program ABI for the full writeup, including a second kernel-side bug (a stack-cost hang in app_restore_after_spawn) found and fixed alongside it.
[partial] Time and OS Services
  Category 4: `time_ticks` used to be a raw `rdtsc` read with no defined relationship to real time; `calibrate_tsc` now measures `rdtsc` ticks per second once at boot by polling the legacy PIT's channel 2 (a known countdown, `rdtsc` before/after, no timer interrupt needed -- this kernel deliberately still has none besides CPU exceptions), averaged across 4 rounds to reduce jitter from a single ~50 ms sample. `time_ticks` (ABI v22) now returns real calibrated nanoseconds since boot. Real wall-clock time also exists now: `time_unix_nanos` reads the CMOS RTC once at boot (BCD-or-binary, 12-or-24-hour, with the classic UIP-wait-then-double-read technique for a torn read) paired with the calibration's boot `rdtsc` snapshot, converted via Howard Hinnant's `days_from_civil` algorithm -- verified live in QEMU against the real host clock (matched the actual date, not just 'a plausible number'). `sleep_nanos` busy-waits against the calibrated clock -- correct, not a placeholder, since RYMOS has no scheduler to hand the CPU to during a sleep anyway. `term_size` reports the console's real row/column count. All four are wired all the way through: kernel ABI, `rymos-user` (`time_ticks`/`time_unix_nanos`/`sleep_nanos`/`term_size`), and `toolchain/rust`'s `std` (`Instant`'s duration arithmetic and `SystemTime` are both real now, no longer honestly-unsupported stubs -- see libstd; `std::thread::sleep` is real too, since sleeping the *current* thread needs no multi-threading support). Verified live in QEMU: `stdshim` sleeps 10 ms and confirms elapsed ticks matches; a `stdreal` `std::thread::sleep(15ms)` measured via real `Instant::checked_duration_since` came back ~15.14 ms; `SystemTime::now().duration_since(UNIX_EPOCH)` reported a plausible real date matching the host. Still open: real terminal/TTY behavior beyond a size query (no raw/cooked mode switching, no ANSI/VT100 escape handling -- left for whenever a concrete consumer needs it, not spec'd out speculatively); sync primitives need no new work (`no_threads` remains the correct answer while RYMOS stays single-threaded per process).
[blocked] Rust Compiler
  Port only after libstd, large persistent filesystem, process spawning, and enough memory are available. libstd's process spawning is now real (see libstd) and stdreal has a repeatable build pipeline, closing the two blockers category 5 had flagged as the last major stretch. A first cargo-shaped smoke test (cargolike/bigoutput, per docs/rust-port-roadmap.md) has now run live too, and found three real gaps stdreal's hand-written test alone hadn't exercised -- a truncating pipe buffer, an env-var ceiling that aborted the whole process, and unwired file mtimes (see Program ABI and libstd) -- all now fixed. A follow-up nested-spawn test (relay) found three more real, layered bugs that would have blocked a genuine cargo port outright (cargo invoking rustc invoking a linker is exactly a nested Command chain) -- pipe-slot exhaustion, a dup2-based stdio-restore bug that silently dropped a nested child's output to the console instead of its caller's pipe, and a kernel-stack-cost bug that hung a deep chain -- all three now fixed and verified 4 levels deep (see Program ABI and libstd). Porting cargo/rustc themselves still hasn't started: this proved the plumbing works end to end on a handful of hand-written test programs, not that cargo's actual dependency graph, build scripts, or rustc's own scale (large files, many descriptors, sustained memory pressure) are dependable yet. Next real step is running larger/further cargo-like helper programs, and eventually a real cargo port, per docs/rust-port-roadmap.md.
